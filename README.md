# coc-star

一个面向国内 TRPG 跑团场景的在线虚拟桌面项目。

本项目从零开始，参考 CCFOLIA 的产品形态与跑团工作流，目标是提供适合中文用户和国内网络环境的在线跑团体验。项目会独立设计和实现，不复制 CCFOLIA 的品牌、素材或源代码。

## 项目定位

CCFOLIA 的核心价值不是单纯提供在线聊天，而是把 GM 在跑团过程中的工作集中到一个可实时协作的房间中：

```text
提前准备场景、地图、角色和演出资源
                  ↓
玩家进入同一个房间
                  ↓
同步查看盘面、角色、道具和状态
                  ↓
通过聊天、角色扮演和骰子推进剧情
                  ↓
GM 快速切换场景、音乐、文本和视觉演出
```

## 参考产品的核心能力

- 房间：一个跑团对应一个独立房间，可邀请成员加入。
- 盘面：支持背景、前景、场景切换、缩放和拖拽。
- 角色：角色立绘、头像、状态、参数、行动顺序和聊天身份。
- 面板：地图、图片、NPC、道具、标记物和可交互元素。
- 卡组：创建牌堆、抽牌、暗牌、查看和回收。
- 聊天：公共频道、信息频道、秘匿聊天、角色发言和聊天记录。
- 骰子：支持常见骰子表达式，并为不同规则系统提供骰子命令。
- 演出：BGM、消息窗口、文字播报、立绘差分和场景切换。
- 权限：GM、协助 GM、玩家、旁观者和禁止访问等房间成员权限。
- 状态管理：保存、加载和恢复跑团过程中的房间状态。

## 第一阶段目标

第一阶段优先验证“多人可以稳定完成一次基础跑团”这条主流程：

1. 创建和加入房间。
2. 在房间内实时同步盘面。
3. 添加、移动和编辑角色棋子。
4. 发送聊天消息并保留聊天记录。
5. 执行基础骰子指令并展示结果。
6. 上传并展示背景、角色和面板图片。
7. 由 GM 切换基础场景。

BGM、复杂卡组、完整角色卡模板、Cut-in、资源市场和高级权限将在基础实时协作稳定后逐步加入。

## 设计原则

- 中文优先：界面、帮助、错误提示和规则说明优先服务中文用户。
- GM 高效：把准备工作沉淀为可复用的场景和资源，减少跑团中的重复操作。
- 实时可靠：房间内的关键状态必须明确同步，避免玩家看到不一致的盘面。
- 规则中立：核心盘面和聊天能力不绑定单一 TRPG 系统，骰子规则通过可扩展方式接入。
- 移动端可用：优先保证手机能够查看、聊天和进行基础操作。
- 独立实现：参考公开产品的功能和用户流程，不直接复制其商标、素材、代码或视觉设计。

## 技术栈

- 前端：React、Vite、TypeScript、Zustand。
- 后端：Python、FastAPI、Pydantic。
- 实时通信：FastAPI WebSocket，后续按并发需要加入 Redis 广播。
- 数据层：PostgreSQL、SQLAlchemy 2、Alembic。
- AI：后端内部独立的 `ai` 领域模块，通过供应商适配器接入不同模型。
- 工具链：pnpm workspace 管理前端，uv 管理 Python，Pytest、Vitest 和 Playwright 负责测试。

详细技术决策见 [docs/architecture.md](docs/architecture.md)。

## 当前状态

项目已完成第一个实时房间闭环：前端可以填写房间号和玩家名，连接 FastAPI WebSocket，查看在线成员并发送基础聊天消息。

当前实时事件包括 `room.connected`、`member.joined`、`member.left`、`chat.message`、`board.token.upserted` 和 `board.token.removed`。角色棋子已经支持创建、拖动同步、删除和数据库恢复；房间已经具备基础的签名会话认证，下一步将完善账号系统和持久化房间权限。
