# 技术栈与架构决策

## 初始技术栈

| 层次 | 选择 | 原因 |
| --- | --- | --- |
| Web 前端 | React + Vite + TypeScript | 适合盘面编辑器、复杂交互和移动端适配，开发反馈快 |
| API 后端 | Python + FastAPI | 异步 WebSocket 支持成熟，便于后续接入 AI、任务队列和数据处理能力 |
| 实时通信 | FastAPI WebSocket | 房间、聊天和盘面同步先保持在同一个后端边界内，减少早期部署复杂度 |
| 数据校验 | Pydantic | 与 FastAPI 集成自然，统一 API 和领域输入校验 |
| 数据库 | PostgreSQL + SQLAlchemy 2 | 支持房间、成员、资源、场景和消息等关系型数据 |
| 数据库迁移 | Alembic（下一阶段接入） | 让领域模型变化可追踪、可回滚；当前初始化阶段使用 `create_all` |
| 缓存/广播 | Redis | 作为多实例部署后的房间广播和短期状态层，初期可暂不启用 |
| 前端状态 | Zustand | 适合盘面局部状态与房间状态，避免过早引入重量级状态框架 |
| 前端样式 | CSS，后续评估 Tailwind CSS | 初始化阶段先保持依赖少，待视觉系统稳定后统一引入原子化样式 |
| 测试 | Pytest + Vitest + Playwright | 分别覆盖 API、前端单元逻辑和真实浏览器流程 |
| 包管理 | pnpm workspace + uv | 分别管理 TypeScript 与 Python 依赖，职责清晰 |

## 目录边界

```text
apps/
  web/                  React 前端与盘面交互
  api/                  FastAPI、房间、聊天和 AI 编排入口
docs/
  architecture.md      当前技术决策
```

后续后端模块建议保持以下边界：

- `room`：房间、成员、权限和连接生命周期。
- `board`：场景、盘面对象、角色棋子和状态变更。
- `chat`：频道、消息、秘匿消息和聊天记录。
- `dice`：骰子表达式解析与规则系统适配。
- `resource`：图片、音频和其他跑团资源。
- `ai`：模型供应商、提示词、上下文和 AI 任务编排。

## 当前身份边界

当前阶段使用服务端签名的房间会话令牌：创建房间签发 GM 令牌，加入已有房间签发玩家令牌，WebSocket 只接受令牌中的身份、房间和角色信息。令牌签名密钥由 `SESSION_SECRET` 提供。

这不是最终账号系统。后续接入账号、登录、刷新令牌和持久化成员权限时，房间 WebSocket 协议可以保持不变，只替换令牌签发与验证来源。

当前已加入账号认证基础层：用户名注册、密码 scrypt 哈希、登录、15 分钟访问令牌和 30 天刷新令牌。现有房间创建/加入流程仍兼容临时房间会话，下一阶段再将房间 API 强制绑定到账号访问令牌。

## AI 接入策略

AI 先作为 API 内部的独立领域模块，不直接散落在房间或聊天代码中。业务模块只依赖统一的 AI 调用接口，具体模型供应商由适配器实现。

第一阶段不把 AI 作为跑团主流程的硬依赖，避免模型不可用时影响房间、聊天和骰子。未来当出现流式输出、长时间任务或高并发需求时，再将 `ai` 拆成独立 worker 或服务。

## 当前取舍

- 暂不使用微服务：项目刚开始，先保持单个 API 服务，降低部署和联调成本。
- 暂不引入消息队列：先通过 WebSocket 完成房间同步；Redis 和任务队列在真实并发需求出现后加入。
- 暂不绑定具体 AI 厂商：先定义供应商无关的接口，方便接入国内模型或自部署模型。
- 暂不建设资源市场：先完成房间内资源上传与使用，再考虑公共资源分发。

## 持久化边界

当前已持久化房间、房间成员和房间棋子。实时房间仍使用内存连接管理器处理广播，数据库负责可恢复的房间状态；服务启动时恢复房间列表，玩家加入时写入成员记录，GM 修改成员角色或移除成员时先提交数据库，再广播实时事件。

本地默认使用 SQLite，正式环境通过 `DATABASE_URL` 使用 PostgreSQL。聊天记录、用户账号和权限数据暂不在本阶段混入，后续分别建立领域模型和迁移。
